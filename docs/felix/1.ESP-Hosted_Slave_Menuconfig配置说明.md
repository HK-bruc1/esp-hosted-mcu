# ESP-Hosted Slave Menuconfig 配置说明

> **源文件版本**：esp-hosted-mcu v2.12.3（本仓库 `slave/main/Kconfig.projbuild` + `slave/main/Kconfig.light_sleep`）

---

## 0. 配置体系概述

在深入具体配置项之前，有必要先理解 Slave 端的配置体系是如何运作的。这能帮助你判断「在哪里改配置」「改哪个文件」「 menuconfig 里的菜单层级是怎么来的」。

### 0.1 Kconfig 与 sdkconfig 的关系

ESP-IDF 使用 **Kconfig** 语言来定义配置菜单，使用 **sdkconfig** 文件来存储实际的配置值。两者的关系可以概括为：

| 概念 | 对应文件 | 作用 |
|------|---------|------|
| **Kconfig（菜单定义）** | `slave/main/Kconfig.projbuild`<br>`slave/main/Kconfig.light_sleep` | 定义了「有哪些选项、数据类型、默认值、依赖关系、菜单层级」 |
| **sdkconfig（配置值）** | `sdkconfig.defaults`<br>`sdkconfig.defaults.esp32c3` 等<br>`sdkconfig`（构建时生成） | 存储「你实际选择了什么值」 |

**数据流向**：

```
Kconfig 源文件  ──→  menuconfig GUI（可视化菜单） ──→  sdkconfig（保存配置值）
                                                          ↓
                                              编译时生成 sdkconfig.h
                                                          ↓
                                              源码通过 CONFIG_XXX 宏引用
```

- `Kconfig.projbuild` 和 `Kconfig.light_sleep` 中的 `config XXX` 定义，在编译后会变成 `sdkconfig.h` 中的 `#define CONFIG_XXX 值`，源码里通过条件编译来使用
- 你在 menuconfig 里勾选或输入的内容，最终都会以 `CONFIG_XXX=值` 的形式写入 `sdkconfig` 文件

### 0.2 slave 目录下 sdkconfig 文件的分工

`slave/` 目录下有多份 `sdkconfig` 相关文件，它们的分工如下：

| 文件 | 作用 | 适用场景 |
|------|------|----------|
| `sdkconfig.defaults` | **所有芯片的通用默认值** | 如 BT 使能、Partition 表选择、编译器优化级别等 |
| `sdkconfig.defaults.esp32` / `esp32c3` / `esp32c6` 等 | **芯片特定覆盖值** | 覆盖该芯片的 GPIO 默认值、CPU 频率、Wi-Fi buffer 大小等 |
| `sdkconfig.ci.*` | **CI 流水线专用** | 只包含最少配置，日常开发不需关心 |
| `sdkconfig.rename` | **配置项重命名映射** | 兼容旧配置名，自动映射到新名 |
| `sdkconfig`（构建生成） | **最终生效配置** | 由 `idf.py menuconfig` 交互修改后保存，通常不提交到版本库 |

**优先级（高 → 低）**：`sdkconfig`（交互修改） > `sdkconfig.defaults.<chip>` > `sdkconfig.defaults`

> **建议**：如果你是做量产固件，建议改 `sdkconfig.defaults.<chip>` 来固化默认配置；如果只是临时调试验证，用 `idf.py menuconfig` 改 `sdkconfig` 即可。

### 0.3 VS Code IDF 插件中的 menuconfig 可视化层级

使用 **VS Code ESP-IDF 插件** 时，可以通过以下方式打开 menuconfig：

- 点击 VS Code **底部状态栏**的 ESP-IDF 图标 → 选择 "Menuconfig"
- 或按 `Ctrl+Shift+P` → 输入 "ESP-IDF: Launch Configuration"

打开后，ESP-Hosted 相关的配置主要分布在两个位置：

**位置一：`Example Configuration` 菜单**
- 这是 `slave/main/Kconfig.projbuild` 定义的内容
- 包含：Bus Config、Wi-Fi、Network Split、Host Power Save、FelixTest 等

**位置二：`Component config → ESP-Hosted` 子菜单**
- 部分更底层的配置（如 Light Sleep）通过 `rsource` 引入
- `Kconfig.light_sleep` 中的内容会显示为 `Light Sleep Power Management` 子菜单

**配置项的可见性规则**：
- Kconfig 中定义了 `depends on` 或 `if` 条件的选项，只有在父选项启用后才会在 GUI 中显示
- 例如：Network Split 的端口配置，只有先勾选 `Enable Network Split` 后才会出现

**Kconfig 语法与 GUI 形态的对应**：

| Kconfig 语法 | menuconfig GUI 形态 |
|-------------|---------------------|
| `config XXX bool` | 复选框（勾选 = y） |
| `config XXX int/string` | 数值/文本输入框 |
| `choice XXX` | 单选下拉框 |
| `menu "子菜单名"` | 可展开的文件夹节点 |
| `depends on XXX` | 父选项未启用时隐藏 |

### 0.4 修改配置的三种方式

| 方式 | 操作 | 影响范围 | 推荐场景 |
|------|------|----------|----------|
| **① menuconfig GUI** | `idf.py menuconfig` 或 VS Code 插件可视化勾选 | 仅当前工程的 `sdkconfig` | 临时调试、探索选项 |
| **② 改 `sdkconfig.defaults.<chip>`** | 编辑文本文件，添加 `CONFIG_XXX=值` | 该芯片的默认构建 | 量产固化、批量同步 |
| **③ 改 `sdkconfig.defaults`** | 编辑文本文件 | 所有芯片的默认构建 | 通用默认值调整 |

> **注意**：方式②和③修改后，建议删除旧的 `sdkconfig` 文件再重新构建，否则旧的 `sdkconfig` 缓存可能覆盖你的新默认值。

## 1. 基础配置

### ESP_HOSTED_COPROCESSOR
- **类型**：bool（隐藏，不可手动选择）
- **默认值**：`y`（始终启用）
- **说明**：ESP-Hosted 协处理器功能的总开关，必须启用

### ESP_HOSTED_CP_BT
- **菜单文本**：`Enable BT sharing via hosted`
- **默认值**：`y`（芯片支持 BT 时），`n`（`!SOC_BT_SUPPORTED` 时）
- **说明**：启用后 Host 可共享协处理器的蓝牙能力。如果 BT 只在协处理器本地使用（不需要 Host 共享），可禁用此选项
- **⚠️ 注意**：如果启用了此选项但未同时启用 `BT_ENABLED` 或 `SOC_BT_SUPPORTED`，menuconfig 会显示警告

### ESP_HOSTED_CP_WIFI
- **菜单文本**：`Enable Wi-Fi on Co-processor`
- **默认值**：`y`
- **依赖**：`SOC_WIFI_SUPPORTED`
- **说明**：启用协处理器上的 Wi-Fi 功能。不需要 Wi-Fi 时可禁用

## 2. Slave 应用模式选择

### SLAVE_APP_MAIN_MODE（choice）
- **菜单文本**：`Slave app_main mode`
- **作用**：选择 Slave 固件的 `app_main()` 入口

| 选项 | 说明 |
|------|------|
| `ESP_HOSTED_COPROCESSOR_APP_MAIN_MODE`（默认） | 使用 ESP-Hosted 内置的协处理器 `app_main()`，正常工作模式 |
| `SLAVE_STANDALONE_TEST_APP_MAIN_MODE` | 独立 SoftAP 测试模式，不等待 Host RPC 命令，仅用于本地 Slave 调试 |

**派生配置**：
- `ESP_HOSTED_COPROCESSOR_APP_MAIN`：`y` if 选择了协处理器模式
- `SLAVE_STANDALONE_TEST_MODE`：`y` if 选择了独立测试模式

## 3. 板级选择

### ESP_HOST_DEV_BOARD（choice）
- **菜单文本**：`ESP PCB Board Selection (optional, used for auto GPIO mapping)`
- **默认值**：`ESP_HOST_DEV_BOARD_NONE`
- **作用**：选择 ESP 开发板型号，自动映射 GPIO 引脚

| 选项 | 说明 | 依赖芯片 |
|------|------|----------|
| `ESP_HOST_DEV_BOARD_NONE`（默认） | 无特定开发板，手动配置 GPIO | — |
| `ESP_HOST_DEV_BOARD_P4_FUNC_BOARD` | ESP32-P4-Function-EV-Board | ESP32-C6 |
| `C2_C5_MODULE_SUB_BOARD` | ESP32-P4 EV Sub-Board（C2 & C5） | ESP32-C2 / C5 |
| `ESP_HOST_DEV_BOARD_P4_C5_CORE` | ESP32-P4-C5 Core Board | ESP32-C5 |
| `ESP_HOST_DEV_BOARD_P4_C6_CORE` | ESP32-P4-C6 Core Board | ESP32-C6 |
| `ESP32P4_EYE_C6_BOARD` | ESP32-P4-EYE Board | ESP32-C6 |
| `ESP_HOST_DEV_BOARD_P4_C61_CORE` | ESP32-P4-C61 Core Board | ESP32-C61 |

> **提示**：选择开发板后，SPI/SDIO/UART/SPI-HD 的 GPIO 引脚和唤醒 GPIO 会自动填充默认值，无需手动配置。

## 4. 测试与调试宏（FelixTest 菜单）

### Standalone 测试模式参数

以下配置仅在 `SLAVE_STANDALONE_TEST_MODE` 启用时可见：

| 配置项 | 说明 | 默认值 |
|--------|------|--------|
| `SLAVE_STANDALONE_TEST_SSID` | SoftAP SSID | `Recorder_AP` |
| `SLAVE_STANDALONE_TEST_PASSWORD` | SoftAP 密码 | `12345678` |
| `SLAVE_STANDALONE_TEST_CHANNEL` | SoftAP 信道（1-13） | `6` |

### SLAVE_SPI_PHY_TEST_DUMP
- **菜单文本**：`Enable SPI full-duplex physical layer TX/RX hex dump`
- **默认值**：`n`
- **依赖**：`ESP_SPI_HOST_INTERFACE`（仅 SPI 全双工模式可见）
- **打印内容**：
  - 原始 SPI TX/RX 缓冲区内容（hex dump 格式）
  - 格式：`spi_tx:` 或 `spi_rx:` 开头，后跟十六进制数据
- **⚠️ 警告**：
  - **会覆盖协议头前 8 字节为测试模式 `0xAA 0x55`**，正常通信会中断
  - 仅在物理层链路验证时使用，正式通信必须关闭

### SLAVE_SPI_PROTO_PKT_TRACE
- **菜单文本**：`Enable SPI protocol packet TX/RX trace`
- **默认值**：`n`
- **依赖**：`ESP_SPI_HOST_INTERFACE`
- **打印内容**：
  - `[SPI_RX_HDR]` — 接收包 header 追踪（校验前）
  - `[SPI_RX_DROP]` — 无效包丢弃原因（零长度/越界/checksum 失败）
  - `[SPI_TX_PKT]` / `[SPI_RX_PKT]` — TX/RX 包摘要（含 seq/checksum）
  - `spi_tx` / `spi_rx` — 完整数据包 hex dump
- **⚠️ 性能影响**：
  - **开启后会显著降低传输速度**（逐包打印开销大）
  - 仅用于调试协议问题，正常使用时必须关闭

### SLAVE_SPI_HD_PHY_TEST_DUMP
- **菜单文本**：`Enable SPI HD / QSPI physical layer TX/RX hex dump`
- **默认值**：`n`
- **依赖**：`ESP_SPI_HD_HOST_INTERFACE`（仅 SPI 半双工模式可见）
- **打印内容**：SPI HD/QSPI 原始 TX/RX 缓冲区内容、控制器回调信息、初始化详情
- **⚠️ 警告**：同 `SLAVE_SPI_PHY_TEST_DUMP`，会覆盖协议头，正常通信中断

## 5. 传输层配置（Bus Config 菜单）

### 5.1 传输层选择

#### ESP_HOST_INTERFACE（choice）
- **菜单文本**：`Transport layer`
- **作用**：选择 Host 与协处理器之间的通信总线

| 选项 | 说明 | 默认条件 |
|------|------|----------|
| `ESP_UART_HOST_INTERFACE` | UART | ESP32-H4 默认 |
| `ESP_SDIO_HOST_INTERFACE` | SDIO | 支持 SDIO Slave 的芯片默认（ESP32、C5、C6、C61） |
| `ESP_SPI_HOST_INTERFACE` | SPI 全双工 | **其他芯片默认**（C2、C3、C5 等） |
| `ESP_SPI_HD_HOST_INTERFACE` | SPI 半双工 | 需手动选择，依赖 `SOC_SPI_SUPPORT_SLAVE_HD_VER2` |

> **Kconfig 默认优先级**：`UART (H4)` → `SDIO (SOC_SDIO_SLAVE_SUPPORTED)` → `SPI FD`

---

### 5.2 SPI 全双工配置

*菜单路径：Bus Config → SPI Full-duplex Configuration*
*依赖：`ESP_SPI_HOST_INTERFACE`*

#### SPI 时钟模式

**ESP_SPI_PRIV_MODE（choice）**

| 选项 | 说明 | 默认 |
|------|------|------|
| `ESP_SPI_PRIV_MODE_0` | CPOL=0, CPHA=0 | — |
| `ESP_SPI_PRIV_MODE_1` | CPOL=0, CPHA=1 | — |
| `ESP_SPI_PRIV_MODE_2` | CPOL=1, CPHA=0 | **ESP32 默认** |
| `ESP_SPI_PRIV_MODE_3` | CPOL=1, CPHA=1 | **其他芯片默认** |

- **派生配置**：`ESP_SPI_MODE` = 0/1/2/3（int，根据选择自动设置）
- **⚠️ 注意**：必须与 Host 端 SPI 模式匹配

#### SPI 控制器选择

**SPI_CONTROLLER（choice）**

| 选项 | 说明 | 默认 |
|------|------|------|
| `SPI_HSPI` | FSPI/HSPI（SPI_controller_1） | **默认** |
| `SPI_VSPI` | VSPI（SPI_controller_2） | 仅 ESP32 支持 |

- **派生配置**：`ESP_SPI_CONTROLLER` = 1（HSPI）或 2（VSPI）

#### GPIO 引脚配置

SPI 全双工需要配置以下引脚。如果你在 §3 选择了某个开发板，这些值会自动填充，无需手动配置。

**基础 SPI 信号**：
- `ESP_SPI_GPIO_MOSI` / `ESP_SPI_GPIO_MISO` / `ESP_SPI_GPIO_CLK` / `ESP_SPI_GPIO_CS`

**控制信号**：
- `ESP_SPI_GPIO_HANDSHAKE` — 握手引脚（Host 查询 Slave 是否就绪）
- `ESP_SPI_GPIO_DATA_READY` — 数据就绪中断引脚（Slave 通知 Host 有数据）
- `ESP_SPI_GPIO_RESET` — 复位引脚（`-1` 表示使用 ESP 自身的 RST/EN 引脚）

> 各芯片的默认 GPIO 值不同（如 ESP32 的 MOSI 默认 13，C2 默认 7），具体数值请在 menuconfig 中查看或查阅 Kconfig 源码。选择开发板后，系统会根据板型自动填充正确的默认值。

#### ESP_SPI_DEASSERT_HS_ON_CS
- **菜单文本**：`Deassert Handshake when SPI CS is deasserted`
- **默认值**：`y`
- **说明**：CS 取消断言后才取消握手并准备新 SPI 事务。防止某些 MCU 在 CS 仍断言时提前开始新事务导致数据丢失

#### 队列大小

**ESP_SPI_TX_Q_SIZE**（ESP → Host 发送队列）

| 芯片 | 默认值 |
|------|--------|
| ESP32 | 10 |
| ESP32-C2（BT 启用） | 6 |
| ESP32-C2 / C61 | 10 |
| 其他（C3, C5, C6, S2, S3 等） | **20** |

- **说明**：队列过小会降低 ESP → SPI → Host 方向的数据速率。增大可提升突发传输性能

**ESP_SPI_RX_Q_SIZE**（Host → ESP 接收队列）

| 芯片 | 默认值 |
|------|--------|
| ESP32 | 10 |
| ESP32-C2（BT 启用） | 6 |
| ESP32-C2（BT 未启用） | **7** |
| ESP32-C61 | 10 |
| 其他 | **20** |

- **说明**：队列过小会降低 Host → SPI → ESP 方向的数据速率

#### ESP_SPI_CHECKSUM
- **菜单文本**：`SPI checksum ENABLE/DISABLE`
- **默认值**：`y`
- **说明**：启用/禁用 SPI 软件校验和。禁用可减少 CPU 开销，但会失去数据完整性保护

### 5.3 SDIO 配置

*菜单路径：Bus Config → SDIO Configuration*
*依赖：`ESP_SDIO_HOST_INTERFACE`*

#### ESP_SDIO_STREAMING_MODE
- **默认值**：`y`
- **说明**：启用 SDIO 流模式，Host 一次性读取队列中的所有数据（而非逐包读取），可提升 Host SDIO 读取性能

#### SDIO GPIO 引脚

SDIO 引脚由芯片硬件固定，**不可软件配置**，menuconfig 中仅作显示参考。ESP32、C5、C6、C61 的引脚各不相同，选择开发板后会自动显示对应的引脚编号。

#### ESP_SDIO_GPIO_RESET
- **默认值**：`-1`（使用 ESP 自身 RST/EN）

#### SDIO 总线速度（choice）

| 选项 | 说明 | 默认 |
|------|------|------|
| `ESP_SDIO_DEFAULT_SPEED` | 20 MHz | — |
| `ESP_SDIO_HIGH_SPEED` | 40 MHz | **默认** |

#### SDIO 从机时序（choice）

| 选项 | 说明 | 默认 |
|------|------|------|
| `ESP_SDIO_PSEND_PSAMPLE` | 发送正沿，采样正沿 | **默认** |
| `ESP_SDIO_NSEND_PSAMPLE` | 发送负沿，采样正沿 | — |
| `ESP_SDIO_PSEND_NSAMPLE` | 发送正沿，采样负沿 | — |
| `ESP_SDIO_NSEND_NSAMPLE` | 发送负沿，采样负沿 | — |

- **说明**：默认时序适用于大多数 SDMMC 控制器，如遇传输错误可尝试其他时序

#### 队列大小

| 配置项 | ESP32-C61 默认 | 其他默认 |
|--------|---------------|---------|
| `ESP_SDIO_TX_Q_SIZE` | 10 | 20 |
| `ESP_SDIO_RX_Q_SIZE` | 10 | 20 |

#### ESP_SDIO_CHECKSUM
- **默认值**：`n`（禁用）

### 5.4 SPI 半双工配置

*菜单路径：Bus Config → SPI Half-duplex Configuration*
*依赖：`ESP_SPI_HD_HOST_INTERFACE`*

#### SPI HD 时钟模式

**ESP_SPI_HD_PRIV_MODE（choice）**
- 选项同 SPI 全双工（Mode 0-3）
- **默认值**：`ESP_SPI_HD_PRIV_MODE_3`
- **派生配置**：`ESP_SPI_HD_MODE` = 0/1/2/3

#### 数据线数量

**ESP_SPI_HD_PRIV_INTERFACE_NUM_DATA_LINES（choice）**

| 选项 | 说明 | 默认 |
|------|------|------|
| `ESP_SPI_HD_PRIV_INTERFACE_4_DATA_LINES` | 4 数据线（QSPI） | **默认** |
| `ESP_SPI_HD_PRIV_INTERFACE_2_DATA_LINES` | 2 数据线 | — |

- **派生配置**：`ESP_SPI_HD_INTERFACE_NUM_DATA_LINES` = 4 或 2

#### GPIO 引脚配置

SPI 半双工需要配置以下引脚。选择开发板后，系统会自动填充对应默认值。

- `ESP_SPI_HD_GPIO_CS` / `ESP_SPI_HD_GPIO_CLK` — 片选和时钟
- `ESP_SPI_HD_GPIO_D0` / `ESP_SPI_HD_GPIO_D1` — 数据线（必配）
- `ESP_SPI_HD_GPIO_D2` / `ESP_SPI_HD_GPIO_D3` — 扩展数据线（4 线模式下使用）
- `ESP_SPI_HD_GPIO_DATA_READY` — 数据就绪中断引脚
- `ESP_SPI_HD_GPIO_RESET` — 复位引脚（`-1` 表示使用 ESP 自身 RST/EN）

#### Data Ready 电平配置（choice）

| 选项 | 说明 | 默认 |
|------|------|------|
| `ESP_DR_ACTIVE_HIGH` | 高电平有效 | **默认** |
| `ESP_DR_ACTIVE_LOW` | 低电平有效 | — |

#### ESP_SPI_HD_Q_SIZE

| 芯片 | 默认值 |

## 6. HCI UART 设置

用于蓝牙 HCI over UART 通信，**不同芯片家族有独立的菜单**。

### 6.1 ESP32-C3 / S3

*依赖：`BT_CTRL_HCI_MODE_UART_H4` 且目标为 ESP32-C3 或 ESP32-S3*

C3 和 S3 的 HCI UART 配置项包括：端口号、波特率、TX/RX 引脚，以及可选的 RTS/CTS 流控制引脚。各芯片的默认引脚不同（如 C3 默认 TX=5，S3 默认 TX=16），具体数值请在 menuconfig 中查看。

- 派生配置：`ESP_HOSTED_BT_UART_FLOWCONTROL_ESP32_C3_S3` = 3（启用流控）或 0（禁用）

### 6.2 ESP32（Classic）

*依赖：`BTDM_CTRL_HCI_MODE_UART_H4` 且目标为 ESP32*

ESP32 的 HCI UART 引脚可在 menuconfig 中直接配置（TX/RX/RTS/CTS）。端口号和波特率则由 Bluetooth Controller 组件决定，在此菜单中只读：

- `ESP_HOSTED_BT_UART_PORT_ESP32` = `BTDM_CTRL_HCI_UART_NO`
- `ESP_HOSTED_BT_UART_BAUDRATE_ESP32` = `BTDM_CTRL_HCI_UART_BAUDRATE`

如需修改端口号/波特率，请前往 `Component config → Bluetooth → Controller Options → HCI Config`。

### 6.3 其他 ESP32 芯片

*依赖：`BT_LE_HCI_INTERFACE_USE_UART` 且非 ESP32/C3/S3/S2*
*适用于：ESP32-C2、C5、C6、C61、H2 等*

所有配置项均为**只读**，值由 Bluetooth Controller 组件 (`BT_LE_HCI_*`) 决定：

| 配置项 | 来源 |
|--------|------|
| `ESP_HOSTED_BT_UART_PORT` | `BT_LE_HCI_UART_PORT` |
| `ESP_HOSTED_BT_UART_TX_PIN` | `BT_LE_HCI_UART_TX_PIN` |
| `ESP_HOSTED_BT_UART_RX_PIN` | `BT_LE_HCI_UART_RX_PIN` |
| `ESP_HOSTED_BT_UART_RTS_PIN` | `BT_LE_HCI_UART_RTS_PIN`（需 `BT_LE_HCI_UART_FLOWCTRL`） |
| `ESP_HOSTED_BT_UART_CTS_PIN` | `BT_LE_HCI_UART_CTS_PIN`（需 `BT_LE_HCI_UART_FLOWCTRL`） |
| `ESP_HOSTED_BT_UART_BAUDRATE` | `BT_LE_HCI_UART_BAUD` |

> **修改方法**：`Component config → Bluetooth → Controller Options → HCI Config`

## 附录 A：GPIO 映射策略

> 来源：`Kconfig.projbuild` 末尾注释

### 跨传输层 GPIO 映射规则

SDIO 引脚作为所有传输层的基准参考，相同物理 GPIO 在不同接口间复用：

| SDIO 信号 | SPI 全双工 | SPI 半双工 | UART |
|-----------|-----------|-----------|------|
| CMD | CS | CS | — |
| CLK | CLK | CLK | — |
| D0 | MOSI | D0 | — |
| D1 | MISO | D1 | — |
| D2 | Handshake | D2 | RX |
| D3 | Data Ready | D3 | TX |

> **注意**：UART Host 侧的 RX 连接 Slave 的 TX，反之亦然

### 资源分配规则

1. **GPIO 有限时**：
   - 仅 1 个额外 GPIO（SDIO 6 个引脚外）：SPI HD 用于 Data Ready；其他模式用于 Host Wakeup（需 RTC-capable GPIO）
   - 有 2 个额外 GPIO：一个 Data Ready，一个 Host Wakeup

2. **深度睡眠要求**：
   - Host Wakeup GPIO **必须**是 RTC-capable GPIO
   - 如果使用非 RTC GPIO（如 P4-EYE 的 IO33），Host 只能从 Light Sleep 唤醒，不能从 Deep Sleep 唤醒

### 开发板示例

| 开发板 | SDIO GPIO | Wakeup |
|--------|-----------|--------|
| ESP32-P4-C6 Core Board | CMD/CLK/D0-D3: 18, 19, 20, 21, 22, 23 | 不可用 |
| ESP32-P4-C61 Core Board | CMD/CLK/D0-D1: 25, 26, 27, 28；D2/D3: 22, 23（UART RX/TX 复用） | 不可用 |

## 附录 C：ESP32-C2 传输效率优化分析

> **背景**：基于 JL BR28 Host + ESP32-C2 Slave SPI 全双工 @24MHz 的 STEP8 UDP 吞吐测试，
> Round 2 实测 117 KB/s (0.96 Mbps)、丢包率 29.6%。以下分析基于 `spi_slave_api.c`、
> `esp_hosted_coprocessor.c`、`slave_wifi_std.c` 源码和 `sdkconfig.defaults.esp32c2` 交叉验证。

### C.1 优化项总览

| 优先级 | 修改项 | 配置位置 | 当前值 | 建议值 | 预期效果 |
|--------|--------|----------|--------|--------|----------|
| **P0** | Wi-Fi 动态 TX buffer | sdkconfig.defaults.esp32c2 | **16** | **32** | 丢包率 29.6% → <10% |
| **P0** | AP_IF TX 重试 | esp_hosted_coprocessor.c（代码） | 无重试 | 2-3次 @1ms | 消除瞬时 Wi-Fi TX 队列满丢包 |
| ~~P1~~ | ~~SPI RX 队列~~ | ~~sdkconfig / Kconfig~~ | **7** | ~~10-12~~ | ⚠️ **已验证：保持默认值**（见 C.5） |
| ~~P1~~ | ~~SPI TX 队列~~ | ~~sdkconfig / Kconfig~~ | 10 | ~~12~~ | ⚠️ **已验证：保持默认值**（见 C.5） |
| **P2** | 编译优化 | sdkconfig.defaults.esp32c2 | SIZE | **PERF** | CPU 热路径提速 5-15% |
| **P2** | SPI 校验和 | menuconfig | y | **n** | 减少 ~1600B/事务的 CPU 开销 |
| **P3** | AMPDU BA WIN | sdkconfig.defaults.esp32c2 | 8 | **16** | Wi-Fi 空口聚合效率提升 |

### C.2 P0-严重：Wi-Fi 动态 TX Buffer = 16（丢包根因）

**问题**：`sdkconfig.defaults.esp32c2` 中：

```
CONFIG_ESP_WIFI_DYNAMIC_TX_BUFFER_NUM=16
```

官方其他芯片默认 32，C2 因 272KB RAM 限制只给了 16。Host STEP8 以 ~122 pkt/s 投喂 SPI，Wi-Fi 空口只能消化 ~86 pkt/s，16 个 buffer 瞬间填满 → `esp_wifi_internal_tx()` 返回错误 → 包被静默丢弃。

**源码路径**（`slave/main/slave_wifi_std.c`）：

```c
#if CONFIG_IDF_TARGET_ESP32C2
    /* Hardcode: No static tx buffers for c2 due to low memory issues */
    if (!cfg.tx_buf_type) {
        cfg.tx_buf_type         = 1;    // Force DYNAMIC
        if (!cfg.dynamic_tx_buf_num)
            cfg.dynamic_tx_buf_num = 16;  // ← 硬编码兜底值
    }
#endif
```

**修改方法**：在 `sdkconfig.defaults.esp32c2` 中：

```diff
- CONFIG_ESP_WIFI_DYNAMIC_TX_BUFFER_NUM=16
+ CONFIG_ESP_WIFI_DYNAMIC_TX_BUFFER_NUM=32
```

**RAM 影响**：每个 dynamic TX buffer ~1.6KB，增加 16 个 ≈ **+25KB**。C2 总 RAM 272KB，需先确认空闲堆（建议在 slave 日志中打印 `heap_caps_get_free_size(MALLOC_CAP_DEFAULT)`）。

### C.3 P0-严重：AP_IF 没有 TX 重试（设计缺陷）

**问题**：STA_IF 有重试逻辑（`MAX_WIFI_STA_TX_RETRY=2`），但 **AP_IF 无重试**。

**源码对比**（`slave/main/esp_hosted_coprocessor.c` `recv_task()`）：

```c
// STA_IF — 有重试 ✓
if (buf_handle->if_type == ESP_STA_IF && station_connected) {
#ifdef CONFIG_ESP_HOSTED_WIFI_TX_RETRY_ENABLED
    do {
        ret = esp_wifi_internal_tx(WIFI_IF_STA, payload, payload_len);
        if (ret) vTaskDelay(pdMS_TO_TICKS(1));
        retry_wifi_tx--;
    } while (ret && retry_wifi_tx);
#else
    ret = esp_wifi_internal_tx(WIFI_IF_STA, payload, payload_len);
#endif

// AP_IF — 无重试 ✗（失败直接丢弃）
} else if (buf_handle->if_type == ESP_AP_IF && softap_started) {
    esp_wifi_internal_tx(WIFI_IF_AP, payload, payload_len);  // 失败就丢
}
```

**影响**：即使 Wi-Fi TX buffer 只是瞬时满（毫秒级），AP_IF 的包也会被丢弃。这在 SoftAP 高吞吐场景下（如 STEP8 UDP 测试）尤为严重。

**修改方法**：需修改 `esp_hosted_coprocessor.c`，给 AP_IF 加入与 STA_IF 相同的重试逻辑：

```c
} else if (buf_handle->if_type == ESP_AP_IF && softap_started) {
#ifdef CONFIG_ESP_HOSTED_WIFI_TX_RETRY_ENABLED
    int retry = MAX_WIFI_STA_TX_RETRY;
    do {
        ret = esp_wifi_internal_tx(WIFI_IF_AP, payload, payload_len);
        if (ret) vTaskDelay(pdMS_TO_TICKS(1));
        retry--;
    } while (ret && retry);
#else
    esp_wifi_internal_tx(WIFI_IF_AP, payload, payload_len);
#endif
}
```

### C.4 流控监测指标错位（架构问题）

**问题**：流控函数 `find_wifi_tx_throttling_to_be_set()` 检查的是 **SPI RX 队列**负载（`spi_rx_queue[PRIO_Q_OTHERS]`），而非 Wi-Fi TX buffer 负载。

**源码**（`slave/main/spi_slave_api.c`）：

```c
static inline int find_wifi_tx_throttling_to_be_set(void)
{
    queue_load = uxQueueMessagesWaiting(spi_rx_queue[PRIO_Q_OTHERS]);
    load_percent = (queue_load * 100 / SPI_RX_QUEUE_SIZE);

    if (load_percent > slv_cfg_g.throttle_high_threshold)  // 监控 SPI RX 队列
        slv_state_g.current_throttling = 1;                 // 而不是 Wi-Fi TX 队列!
    // ...
}
```

**结果**：
- SPI RX 队列（容量 7）交易速度匹配，从不拥塞 → **流控永远不触发**
- Wi-Fi TX buffer 溢出时，流控机制**完全无感知**
- 这解释了 Round 2 日志 `flw_ctrl on=0` 的根因

**结论**：这是 esp-hosted-mcu **官方架构设计**，menuconfig 无法解决。改进方案需要让流控也感知 Wi-Fi TX 队列深度，可考虑在 throttle 检查中增加 `esp_wifi_internal_tx_is_stop()` 查询（ESP-IDF 内部函数，可获取 Wi-Fi TX buffer 是否已满）。

### C.5 ~~P1~~ ：SPI 队列大小 — 实测结论：保持默认值

**原始问题**：C2 的 `ESP_SPI_RX_Q_SIZE` 默认 7（所有芯片中最小），理论上是 Host→ESP 方向的瓶颈。

**实测验证（2026-04-16）**：

在 menuconfig 中调整 `ESP to Host SPI queue size` 和 `Host to ESP SPI queue size`，测试结果：

| 配置 | TX Q | RX Q | 结果 |
|------|------|------|------|
| 默认值 | 10 | 7 | ✅ 正常工作 |
| 适度增大 | 12 | 10 | ✅ 正常工作 |
| 极限值 | 20 | 20 | ✅ 正常工作（极限） |
| 超过 20 | >20 | >20 | ❌ **异常**（内存不足或行为异常） |

**结论：C2 的 SPI 队列极限为 20，超过即异常。** 这是因为队列大小直接决定内存池块数：

```c
// spi_slave_api.c:141
#define SPI_MEMPOOL_NUM_BLOCKS  ((SPI_TX_QUEUE_SIZE+SPI_RX_QUEUE_SIZE)+SPI_QUEUE_SIZE*2)
```

每个 block = 1600 字节，队列越大 → 预分配的 SPI buffer 越多 → C2 的 272KB RAM 迅速耗尽。

**最终决定：保持 Kconfig 默认值（TX=10, RX=7），不在 sdkconfig.defaults.esp32c2 中覆盖。** 原因：
1. C2 内存极度紧张，队列增大的 RAM 开销远超收益
2. SPI 队列不是当前吞吐瓶颈（瓶颈在 TCP 窗口和 Wi-Fi TX buffer）
3. 超过 20 就会异常，安全余量不足

> **menuconfig 与 sdkconfig 的关系**：menuconfig 中的 `ESP to Host SPI queue size` 就是 `CONFIG_ESP_SPI_TX_Q_SIZE`，`Host to ESP SPI queue size` 就是 `CONFIG_ESP_SPI_RX_Q_SIZE`。它们最终映射到 `spi_slave_api.c` 中的 `SPI_TX_QUEUE_SIZE` 和 `SPI_RX_QUEUE_SIZE` 宏，控制 FreeRTOS 队列深度。在 menuconfig 里改和在 sdkconfig.defaults 里写效果完全一样。

### C.6 P2：编译优化 SIZE → PERF

**当前**：

```
CONFIG_COMPILER_OPTIMIZATION_SIZE=y    # 当前：最小体积
#CONFIG_COMPILER_OPTIMIZATION_PERF=y   # 注释掉的：最大性能
```

**影响**：SIZE 优化会导致热路径函数（SPI 事务循环、`esp_wifi_internal_tx()`）未被 inline 展开，分支预测提示缺失。对 120MHz 单核 C2 CPU 影响显著。

**权衡**：PERF 会增加 Flash 占用（~10-20%）。C2 配置为 2MB Flash，需确认空间足够。

### C.7 P2：禁用 SPI 校验和

**当前**：`ESP_SPI_CHECKSUM=y`（默认启用）

每笔 1600 字节 SPI 事务都做软件校验和，对 120MHz C2 是非零 CPU 开销。如果 SPI 信号完整性已验证稳定（24MHz 短距离 PCB 连线），可禁用。

**修改方法**：menuconfig 中 `SPI checksum ENABLE/DISABLE` → 禁用，或：

```
CONFIG_ESP_SPI_CHECKSUM=n
```

### C.8 P3：AMPDU BA WIN

**当前**：

```
CONFIG_ESP_WIFI_TX_BA_WIN=8
CONFIG_ESP_WIFI_RX_BA_WIN=8
```

AMPDU 聚合窗口越大，Wi-Fi 每次聚合的帧越多，空口效率越高。8 是保守值。

**建议**：`8` → `16`（如 RAM 允许）

## 附录 D：关键实测结论汇总（2026-04-16）

### D.1 SPI 队列大小（menuconfig）与源码的映射关系

menuconfig 中的配置项与代码的对应关系：

```
menuconfig "ESP to Host SPI queue size"    →  CONFIG_ESP_SPI_TX_Q_SIZE
    → spi_slave_api.c: #define SPI_TX_QUEUE_SIZE  CONFIG_ESP_SPI_TX_Q_SIZE
    → spi_tx_queue[] (FreeRTOS Queue)

menuconfig "Host to ESP SPI queue size"    →  CONFIG_ESP_SPI_RX_Q_SIZE
    → spi_slave_api.c: #define SPI_RX_QUEUE_SIZE  CONFIG_ESP_SPI_RX_Q_SIZE
    → spi_rx_queue[] (FreeRTOS Queue)
```

**在 menuconfig 里修改和在 `sdkconfig.defaults.esp32c2` 中写配置行效果完全一样。**

队列大小还决定了内存池预分配块数：
```c
#define SPI_MEMPOOL_NUM_BLOCKS  ((SPI_TX_QUEUE_SIZE + SPI_RX_QUEUE_SIZE) + SPI_QUEUE_SIZE * 2)
```
每块 1600 字节。C2 上实测极限为 TX+RX 合计 ≤ 40（各 20），超过即内存不足导致异常。

### D.2 RawTP（Raw Throughput Test）不能测 Wi-Fi AP 吞吐

`RawTP: Transport level throughput debug test` 启用后速率始终为 0 的原因：

1. **RawTP 测的是纯 SPI 总线速率，不经过 Wi-Fi**
   - TX 方向：`raw_tp_tx_task()` 直接向 `send_to_host_queue()` 发送 `ESP_TEST_IF` 类型测试数据（`0xdeadbeef` 填充）
   - RX 方向：`process_rx_pkt()` 收到 `ESP_TEST_IF` 包只更新计数器 `debug_update_raw_tp_rx_count()`
   - **两个方向都完全不经过 Wi-Fi 驱动**

2. **速率为 0 的根因：需要 Host 端同时启用 RawTP**
   - Slave 启用后，在能力协商阶段告知 Host 自己支持 RawTP
   - **Host 必须回传 `SLV_CONFIG_TEST_RAW_TP` 配置**，Slave 收到后才会调用 `process_test_capabilities()` 创建 `raw_tp_tx_task`
   - 如果 Host（如 JL BR28）未启用 RawTP，`raw_tp_tx_task` 永远不会被创建，RX 也收不到测试包 → 双向速率 = 0

3. **正确使用方式**：Slave 和 Host **双端**都启用 RawTP，测试的是 SPI 传输层的极限带宽，用于验证 SPI 链路移植是否正常

**如果要测 Wi-Fi AP 方向吞吐，应使用**：
- `ESP_PKT_STATS`（传输层包统计，已启用）
- iperf 工具（Wi-Fi 客户端连 SoftAP）
- 应用层测试（如 STEP7/STEP8 的 `sock_send` 速率统计）
